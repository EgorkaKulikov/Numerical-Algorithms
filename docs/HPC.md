# Вычислительная эффективность и HPC-готовность

Документ описывает, как кодовая база адаптирована под требование о вычислительной
эффективности и пригодности для высокопроизводительных вычислительных сред.

## Целевой режим

Плотные матрицы, размер системы `dim = n + 2` (n — число узлов сетки), N до ~10⁴.
Узкие места: (1) сборка матриц проекционно-коллокационных схем, (2) плотная
линейная алгебра O(N³).

## Архитектура ускорения (два уровня)

1. **Линейная алгебра → нативный BLAS.** `numerics.LinearAlgebra` сохраняет
   прежний массивный API (`Array<DoubleArray>`/`DoubleArray`), но тяжёлые операции
   (`matVec`, `matTransVec`, `matMat`, `atWa`, `addScaled`, `solve`) делегируются
   библиотеке **multik** с движком **OpenBLAS** (`multik-default`). Решение СЛАУ
   O(N³) выполняется нативным LAPACK с внутренней многопоточностью.
   Эталоном корректности служит `numerics.ReferenceLinearAlgebra` (рукописная
   реализация), с которой результаты сверяются в тестах.

2. **Сборка матриц → многоядерный JVM-параллелизм.** `numerics.ParallelAssembly`
   распределяет независимые строки/столбцы матриц по ядрам через
   `java.util.stream.IntStream.parallel()` (общий ForkJoinPool, без сторонних
   зависимостей). Каждая задача пишет в свою строку → отсутствие гонок.
   Подключено в горячих местах: `matrixM`/`matrixM2` (Fredholm, Volterra),
   B-матрица и якобианы Ньютона/Кулкарни (Uryson).

   Намеренно оставлены последовательными (документировано в коде): симметричная
   сборка матрицы Грама (перекрёстные записи), scatter-add аккумуляция, и
   конечно-разностный якобиан Ньютона-Нюстрёма (общий изменяемый вектор `x`).

### Семантика сохранена

- `solve` бросает `IllegalStateException` на вырожденной матрице (LAPACK может
  вернуть NaN — это проверяется явно), входные данные не мутируются.
- `cholesky` остаётся health-check'ом (возвращает `L` для ПОО или `null`).
- Все наблюдаемые выходы решателей (`E_h`, конечность) защищены сетью из 124
  тестов, включая характеризационные golden-тесты трёх решателей.

## Подключаемый бэкенд (SPI) — готовность к другим HPC-средам

Тяжёлые операции линейной алгебры вынесены за интерфейс
`numerics.backend.LinAlgBackend` (Service Provider Interface). Реализации
регистрируются в `numerics.backend.Backends`, а `LinearAlgebra` — тонкий фасад,
делегирующий активному бэкенду. Поставляются два рабочих бэкенда:

| Бэкенд | Описание | Доступность |
|--------|----------|-------------|
| `MultikCpuBackend` | нативный multik/OpenBLAS (по умолчанию) | реальная проба `mk.linalg.solve` |
| `ReferenceBackend` | чистый JVM (эталон/фолбэк) | всегда |

`Backends` выбирает первый доступный бэкенд (можно переопределить свойством
`-Dnumerics.backend=multik|reference`) и **автоматически откатывается** на
`ReferenceBackend`, если нативная библиотека OpenBLAS недоступна. Это и есть
переносимость для HPC: предпочесть ускоренный бэкенд, но гарантировать
работоспособность на любом узле.

### GPU — точка расширения, не реализация

GPU-бэкенд добавляется без изменения кода решателей и фасада: достаточно
реализовать `LinAlgBackend` (например, поверх JCublas/cuSOLVER на NVIDIA-узле
или Metal на Apple Silicon) и зарегистрировать его в `Backends`. На текущей
машине (Apple Silicon, без CUDA) и в самой multik GPU-движка нет, поэтому GPU
оставлен как реализация интерфейса для будущего HPC-узла с подходящим железом —
архитектурно код к этому готов.

## Бенчмарк

```
./gradlew runBenchmark
```

Выводит две таблицы:
- (A) масштабирование: медианное/мин. время полного решения Fredholm F2 vs N;
- (B) ускорение параллельной сборки матрицы относительно последовательной + число ядер.

Прогрев JIT/OpenBLAS и медиана по повторам включены. Пример (arm64 macOS, 8 ядер,
JDK 21): полный решатель n=256 (dim=258) ~120 мс; ускорение сборки матрицы
~4–5.7x на 8 ядрах (ограничено стоимостью квадратуры на ячейку и пропускной
способностью памяти при мелкозернистых задачах).
